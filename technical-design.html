Development Tools

The game is developed using JavaScript on the client and on the server through Node.js. Google Chrome is the project’s main supported browser.

Visual Studio Code has been used extensively during development of both client and server software. The project makes use of Git for version control, with both client and server part of the same repository to ensure that they are kept in sync. NPM is used to fetch and install the project’s dependencies.

The project depends upon numerous third party libraries. On the client, Phaser 3 is used for input handling and rendering. Socket.io is used for both client-side and server-side networking. The Express.js framework is used to create the web server. The node-gameloop is used to create an efficient main application loop in our Node.js application. To process the level geometry at runtime, the jsClipper library is used along with Clipsy and MarchingSquaresJS on the server. Pngjs is used for loading images on the server and Planck.js is used as the game’s physics library.

General Constraints

There are several limitations and constraints that have impacted the design of the project. Time is certainly the biggest factor, alongside software availability. The project had to be completed in a little over two weeks due to the development team’s schedules. The project team consists of two people, meaning careful time management and gameplay design choices were necessary to ensure the project could be delivered.

Code Style

The codebase uses some ECMAScript 6 (ES6) features. The ES6 style of development is well documented by Phaser, so we decided to implement our game’s scenes using classes. The client has been tested extensively on Chrome (version 73).

Development Methodology/Planning

Software development methodologies encourage a systemic approach towards development to improve the quality of the final product. There was not a lot of time to create a forward plan for this project. This meant that the project was implemented alongside additional research and experimentation, with the design and requirements evolving over development. This means that a traditional linear project management methodology, such as the Waterfall model, was unsuitable. Rewriting the project schedule to compensate for new obstacles would have been very time consuming, which was unacceptable given the time constraints the project already faced.

While the project did not benefit from a rigid linear project plan, deadlines still needed to be met. Development was broken into discrete tasks that could be completed over before moving to the next issue. To ensure that the project was progressing as planned, the project was reassessed at the end of each task. The development style could be described as a loose style of agile development.
